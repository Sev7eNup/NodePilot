02

Was ist NodePilot?

NodePilot ist eine agentless Workflow-Orchestrierung für Windows-Infrastrukturen. Statt Agenten auf jedem Zielsystem zu installieren, spricht NodePilot Maschinen über WinRM an.

Es ist der moderne, schlanke Nachfolger des Microsoft System Center Orchestrator — gleiche Konzepte (Runbooks, Activities, Schedules, Edit-Lock), aber auf aktuellem Stack gebaut.

🪶
Schlank
Ein Windows Service, eine API, eine SPA
🔌
Agentless
Keine Agent-Installation auf Zielen
⚡
Echtzeit
SignalR-Streams während Ausführung
🧩
Erweiterbar
Custom Activities, MCP, KI-Assistent

Activity-Konfiguration

Pro Step
config.retry
maxAttempts · backoff · initialDelayMs · maxDelayMs
config.timeoutSeconds
per-Step Timeout — zusätzlich zum Execution-Timeout
config.isolated: true
runScript nur lokal · Job Object (KILL_ON_JOB_CLOSE) · memoryLimitMb + maxProcesses
successExitCodes
Opt-in z. B. "0,1" → non-zero Exit wird zum Fehler
Erfolgssemantik (runScript)

Ein Step scheitert nur bei einem terminierenden PowerShell-Fehler (throw / Write-Error unter Stop-Wrapper). Ein exit N macht den Step nicht rot.

# exit N macht den Step NICHT rot
exit 1 # → success: true, exitCode: 1
throw "boom" # → success: false
Exit-Code liegt immer unter {{step.param.exitCode}}.

Workflow-JSON Format

Struktur
{
  "nodes": [{
    "id": "step-123",
    "type": "activity",
    "data": {
      "label": "Check Disk",
      "activityType": "runScript",
      "targetMachineId": "guid",
      "outputVariable": "diskCheck",
      "config": { "script": "Get-PSDrive C" }
    }
  }],
  "edges": [{
    "source": "step-123",
    "target": "step-456",
    "data": { "condition": "step-123.success" }
  }]
}
Nodes

Ein Node = eine Activity-Ausführung. activityType → Executor, config → Parameter.

Edges

Source → Target. Tragen condition + optionale controlPoints.

Layout

data.controlPoints überschreibt Auto-Routing. Styleguide: docs/workflow-styleguide.md.

21

Edge Conditions

Shortcut-Form
"step-123.success" // bei Erfolg
"step-123.failed" // bei Fehlschlag
null // immer
disabled: true // übersprungen
Expression-AST (conditionExpression)
comparison — ==, !=, <, contains, matches, isEmpty …
group — AND / OR · not — Negation
Operanden: variable oder literal
{ "type": "comparison", "left": {"type":"variable","name":"disk.free"}, "op": "<", "right": {"type":"literal","value":"10"} }

Disabled Edges: Target-Node wird nicht Root. Alle eingehenden disabled → Node wird Skipped.

Databus / Variablenauflösung

Auflösungs-Tails
{{varName.output}} // Stdout
{{varName.error}} // Stderr
{{varName.success}} // "true" / "false"
{{varName.param.X}} // OutputParameter
{{globals.NAME}} // Globale Variable

Kein outputVariable → Step-ID wird genutzt.

Strukturierter Output (runScript)

Deklarierte Variablen werden als param.* gecaptured:

$hostName = "SRV01"
# → {{step.param.hostName}} = "SRV01"
Auto-Quoting

{{step.output}} wird als Single-Quoted String eingesetzt. Im Script $x = {{step.output}} — nicht extra quoten.

Contract-Garantie

Nur output/error/param.X werden aufgelöst — andere Tails bleiben Literal.

Edit-Lifecycle (Edit-Lock)

Per-User-Edit-Lock (CheckedOutByUserId). Mutierende Endpoints → 423 Locked wenn Caller nicht Lock-Owner.

🔒
POST /lock
Atomar IsEnabled=false + Lock setzen
🔓
POST /unlock
Lock-Fields auf null
🚀
POST /publish
Save + Enable + Unlock (atomar)
🛡️
POST /force-unlock
Admin-only — bricht fremden Lock
Button-State-Matrix
canWrite = role !== 'Viewer' && checkedOutByUserId === currentUserId

Disable ist nicht lock-gegated (Incident-Kill-Switch). Enable braucht keinen Lock (423 sonst).

Custom Activities — Plugin-System

Was?

User-authored, wiederverwendbare Activities — parametrisiertes PowerShell-Template. Ein runScript-Preset, keine zweite Script-Engine.

Dispatch

activityType = custom:<key> → CustomActivityExecutor. Definition im Config (__customDefinitionId + __customKey Drift-Guard).

Governance
  • • Create/Edit/Delete = Admin + Operator, nur solange disabled
  • • Enable/Disable + Mutation enabled Defs = Admin-only
  • • Optimistic Concurrency via ConcurrencyToken → 409
  • • Execution speichert CustomActivity{Key,Version,Hash}
  • • Import landet disabled (Admin-Review)

Best Practices

Workflows
  • • Styleguide docs/workflow-styleguide.md zuerst

  • • outputVariable sprechend benennen
  • • Edges mit Conditions, nicht mit Reihenfolge-Annahmen

  • • Secrets via {{globals.X}} / Credentials
Engine & Operations
  • • throw für Fehler, nicht exit N
  • • Lokales runScript mit isolated: true
  • • Sub-Workflows für Wiederverwendung, Max-Depth 10
  • • Retention-Services konfigurieren
  • • Alerting opt-in per Daten — idle bis Regel existiert
  • • Tests sind Pflicht (Method_Scenario_Expected)
